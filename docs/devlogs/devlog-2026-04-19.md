## 1. Trivy Vulnerability Scan

Installed Trivy on the Ubuntu VM via the Aqua Security apt repository. Ran initial scan against `auxcon/labwatch-api:0.1.0`.

**Initial findings:** 47 OS findings, 3 Python dependency findings. No CRITICAL. No secrets detected.

**Triage:**

The 3 Python findings were immediately actionable — both had fixed versions available and were in our own `requirements.txt`:

- `requests 2.32.3` — CVE-2024-47081 (.netrc credential leak via redirect) and CVE-2026-25645 (predictable temp file creation). Fixed by bumping to `2.33.0`.
- `pip 25.0.1` (system pip in base image) — CVE-2025-8869 (symlink extraction bypass). Fixed by adding `pip install --upgrade pip` to the runtime stage of the Dockerfile.

One HIGH OS finding was fixable:

- `libssl3t64 / openssl` — CVE-2026-28390 (null pointer dereference in CMS, DoS). Fixed version `3.5.5-1~deb13u2` was available in the Debian security repo. Fixed by adding `apt-get upgrade -y --no-install-recommends` to the Dockerfile runtime stage.

**Remaining 32 findings after remediation** are all upstream-blocked with no fixed version. Key ones documented:

- CVE-2025-69720 (ncurses, HIGH x4) — terminal UI buffer overflow. ncurses is never invoked at runtime in a headless HTTP API. Negligible risk.
- CVE-2026-29111 (systemd, HIGH x2) — code execution via spurious IPC. systemd is not running as PID 1 in the container. libsystemd0 is a library on disk only. Attack requires active D-Bus connections. Negligible risk.
- Remaining MEDIUM findings affect glibc, util-linux, xz, zlib, tar, systemd nspawn/udev. Attack vectors require local system access or physical hardware — not exposed by a Flask API over port 8080.

**Outcome:** SECURITY.md written and committed documenting all findings, risk reasoning, and remediation policy. Before/after scan artifacts saved to `security-reports/`. A peer running Grype or Trivy against `0.2.0` will find the same 32 findings already documented with reasoning.

## 2. README IP Cleanup

The architecture diagram in README.md contained specific internal IPs for all lab devices, effectively publishing the network topology in a public repo. This was identified as an OPSEC issue — the router and main PC were already excluded from `config.py` for the same reason, so the README should follow the same policy.

Replaced all internal IPs with role labels:

- `192.168.x.x` → `[HOME-ROUTER]`
- `192.168.x.x` → `[CORE-SWITCH]`
- `192.168.x.x` → `[DEV-MACHINE]`
- `192.168.x.x` → `[LAB-PC]`
- `192.168.x.x` → `[PROD-SERVER]`

Also sanitised the example JSON response in the API section — the `target` field for a ping check was showing a real LAN IP, replaced with `[LAN-HOST]`.

The topology diagram remains fully readable for the instructor and peers. No information about the architecture is lost — only the specific IPs are removed.

## 3. GitHub Actions CI/CD

**Problem being solved:** The build process was entirely manual. Changes required SSH into the VM, editing files, rebuilding the container, and running `docker buildx build` by hand. Two machines (Windows dev, Ubuntu VM) had separate copies of the repo that drifted out of sync — this caused a broken build earlier in the session where the VM rebuilt from stale files and the fixes didn't land.

**Workflow implemented:** `.github/workflows/docker-build.yml`

- Trigger: push to `main` branch
- Runner: `ubuntu-latest` (GitHub-hosted)
- Steps: checkout → setup buildx → DockerHub login → multi-platform build and push
- Platforms: `linux/amd64`, `linux/arm64`
- Tags: `auxcon/labwatch-api:0.2.0`, `auxcon/labwatch-api:latest`
- Build context: `./backend`

**Secrets configured in GitHub repo settings:**
- `DOCKERHUB_USERNAME` — DockerHub account `auxcon`
- `DOCKERHUB_TOKEN` — dedicated PAT named `github-actions` with Read+Write scope (separate from the Docker Desktop auto-generated token)

**Issues encountered during setup:**

*Secret misconfiguration* — Initial run failed with "Username and password required". Root cause: both secret names were pasted into the value field of a single secret named `AUXCON` instead of creating two separate secrets. Corrected by deleting the combined secret and creating two properly named ones.

*Wrong build context* — Second run failed: `"/app": not found`. The workflow had `context: .` (repo root) but the Dockerfile, app code, config, and gunicorn config all live in `backend/`. Fixed by changing to `context: ./backend`.

*Stale app code in repo* — Third run completed the build but the container crash-looped with `ImportError: cannot import name 'check_all_services' from 'app.checker'`. Root cause: `backend/app/checker.py` in the repo was an old version with only one function (`check_service`). The working version in `monitoring-api/app/checker.py` has four functions. The repo had never been properly synced with the actual working code. Fixed by copying the correct `checker.py`, `main.py`, `config.py`, and `gunicorn.conf.py` from `monitoring-api/` into `backend/` and committing.

**Final pipeline result:** Build ran in 1m 51s. `0.2.0` and `latest` tags pushed to DockerHub for both `linux/amd64` and `linux/arm64`. Image size increased from ~48MB (0.1.0) to ~58MB (0.2.0) — expected, due to `apt-get upgrade` pulling patched packages.

## Infrastructure Issues Discovered

**Two-copy problem:** The working production code lives in `/home/nate/lab/monitoring-api/` on the Ubuntu VM. This directory is not a git repository and never has been — files were deployed manually at some point before this session. The git repo clone at `/home/nate/lab/home-lab-monitoring-project/` is a separate copy that was significantly out of sync with the working deployment.

This caused repeated issues throughout the session: Dockerfile edits not taking effect, stale `checker.py` breaking the CI build, git operations being run in the wrong directory. The two directories were manually synced during this session but will drift again.

**Resolution plan (Phase 7):** Replace `monitoring-api/` with a proper deployment that pulls from the repo. The Kubernetes phase will restructure the deployment so the VM runs from the repo clone directly, eliminating the manual copy problem.

**Redundant root-level files:** As a side effect of today's manual syncing, `Dockerfile`, `requirements.txt`, and `SECURITY.md` now exist at both the repo root and in `backend/`. The root copies are redundant. These will be removed as part of the pre-Kubernetes cleanup.

## Git Configuration Applied

Both machines were missing some git configuration that caused friction:

- `git config --global pull.rebase true` — set on both Windows and Ubuntu VM to prevent divergent branch errors
- `git config --global core.autocrlf false` — set on Windows to prevent CRLF line ending pollution
- `git config --global credential.helper store` — set on Ubuntu VM to cache the GitHub PAT and avoid repeated prompts
- `git config --global user.name` and `user.email` — configured on Ubuntu VM (had not been set)
