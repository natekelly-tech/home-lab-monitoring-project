### Unit 3 -- Syslog and Log Management Review

Full review of rsyslog configuration, logrotate configuration, time synchronization, and cross-layer log timestamp consistency. Findings documented below.

**rsyslog**

- Version: 8.2312.0 (aka 2023.12), compiled with systemd support
- Status: active (running) since 2026-04-21 17:00:40 UTC
- Received SIGHUP at 2026-04-26 00:00:03 UTC from logrotate -- correct behavior, confirms log rotation and file handle reload are working end to end
- Config: /etc/rsyslog.conf loads imuxsock (local socket) and imklog (kernel log) modules. Rules live in /etc/rsyslog.d/50-default.conf. Three drop-ins active: 20-ufw.conf, 21-cloudinit.conf, 50-default.conf.

Active logging rules (50-default.conf):
- `auth,authpriv.* -> /var/log/auth.log` -- captures all SSH logins, sudo invocations, and PAM authentication events. fail2ban reads from this file for its SSH jail.
- `*.*;auth,authpriv.none -> /var/log/syslog` -- general system events. k3s logs directly to syslog -- Kubernetes control plane activity (CIDR allocation, load balancer updates, etcd compaction) is captured at the OS log layer in addition to kubectl logs.
- `kern.* -> /var/log/kern.log` -- direct kernel messages including UFW BLOCK entries.

Evidence -- SSH session captured in auth.log: sudo commands logged with full command path, working directory, and microsecond-precision UTC timestamps. Example from this session: ubuntu invoked /usr/bin/systemctl status rsyslog at 2026-04-26T11:53:29 UTC. Audit trail is functioning correctly.

Notable syslog entries observed:
- UFW BLOCK entries for ICMPv6 Type 133 (router solicitation) on cni0 interface. Expected behavior. k3s pods send router solicitation messages on the container network bridge. UFW blocks them because they are not in the allowlist. Not an attack indicator.
- amazon-ssm-agent logging credential rotation every 30 minutes. SSM agent is active, relevant when CloudWatch access is provisioned by Dollahite.

**logrotate**

Configuration: /etc/logrotate.d/rsyslog
- Files managed: syslog, mail.log, kern.log, auth.log, user.log, cron.log
- Frequency: weekly
- Retention: 4 rotations (4 weeks of history)
- `compress` + `delaycompress`: rotated files are gzipped, but the most recent rotation is held uncompressed for one cycle so tools with open file handles (e.g. fail2ban) do not encounter unexpected gzip streams
- `missingok` + `notifempty`: skips silently if a log file is missing or empty
- `postrotate`: calls /usr/lib/rsyslog/rsyslog-rotate, which sends SIGHUP to rsyslogd to reopen file handles after rotation. Confirmed working -- SIGHUP visible in systemctl status output at 2026-04-26 00:00 UTC.
- `sharedscripts`: postrotate runs once after all files rotate, not once per file.

logrotate also manages: fail2ban, certbot, ufw, unattended-upgrades. Log management covers all security tooling, not just rsyslog.

Dry run result: no errors. All rsyslog-managed files reported "log does not need rotating" -- correct, as rotation ran at midnight.

**Known gap -- Gunicorn access logs not in structured JSON format**

The application logger configured in main.py emits JSON with a timestamp field via python-json-logger's rename_fields. However, Gunicorn produces its own HTTP access logs (remote_ip, method, path, status, duration_ms, bytes) through a separate logging path that bypasses the application's JsonFormatter. These access logs use Gunicorn's default Apache-style format, not JSON, which limits structured log analysis and field-level querying. Reconfiguring Gunicorn to route access logs through the JsonFormatter (typically via a gunicorn.conf.py) is tracked as a Phase 8 improvement item and is the substantive fix scheduled for the next session's priority 2.

## Issues Encountered and Resolved

### cloudflared reload not supported
`systemctl reload cloudflared` returned: Job type reload is not applicable for unit cloudflared.service. Full restart used instead. Brief tunnel drop of a few seconds during reconnect. Not zero-downtime but acceptable for a lab environment.

### Stale k3s pod image
Pod had been running since 2026-04-22 on the pre-v0.4.1 image. `kubectl rollout restart` forces a fresh pull without changing the manifest. This is the correct approach when the image tag (latest) has not changed but the underlying image has been updated.

---

## Decisions Made

### Tunnel routes to NodePort, not pod IP
Pod IPs change on every restart. The k3s Service provides a stable NodePort (31012) on the host. Cloudflare Tunnel targets the NodePort, not the pod directly. This is the correct separation -- the tunnel does not need to track ephemeral pod IPs. The Service handles routing to whatever pod is currently healthy.

### Docker Compose stopped on EC2 but not removed
Docker remains installed on EC2. The docker-compose.yml is retained in /home/ubuntu/lab/ as a reference and fallback. If k3s were to have a critical failure, Docker Compose could restart the app in under a minute. This is a reasonable contingency for a single-node cluster with no secondary node to reschedule pods to.
