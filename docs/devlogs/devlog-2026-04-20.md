## Summary

Today was a significant operational milestone. LabWatch moved from a home Ubuntu VM in Germany to an AWS EC2 instance in us-west-1. The production API is now running 24/7 independent of home infrastructure, accessible to instructors and peers at any hour regardless of time zone.

The migration was driven by two concrete operational problems: a 9-hour time difference between the developer (Germany) and the instructor (US West Coast) making home lab uptime incompatible with a production system, and a red-team exercise scheduled this week that raised concerns about home network exposure if a compromised lab machine could reach personal devices on the same flat subnet.

## What Was Done

### Security Assessment (Pre-Migration)

Reviewed a video covering Linux VPS hardening techniques. Cross-referenced the recommendations against the existing home lab setup. Identified that the primary risk was not the Ubuntu VM itself but the flat /24 network — all devices including personal PCs shared the same subnet with no VLAN isolation. A compromised lab machine could reach personal devices directly.

Options evaluated: VLAN segmentation on the Cisco Catalyst 1200 (correct engineering answer, longer timeline), or cloud migration (faster, removes risk entirely before red-team). Given the red-team timeline, cloud was the right call.

## Decisions Made

**Why EC2 over VLAN segmentation:** VLAN configuration on the Cisco Catalyst is the architecturally correct long-term answer and will still be done as a learning exercise. However the red-team exercise is this week. Configuring VLANs incorrectly under time pressure risks either misconfiguring the network or still having a window of exposure. EC2 eliminates the risk entirely and immediately. The home lab remains available for personal projects and future experiments.

**Why reuse the existing Cloudflare Tunnel:** Creating a new tunnel would require updating Cloudflare DNS records. Reusing the existing tunnel with new connector credentials is cleaner — the DNS record does not change, the domain stays the same, and the transition is invisible to end users.

**Why latest tag on EC2:** Pinned tags are correct for reproducible builds. However EC2 is now the deployment target and needs to track releases. Using latest means `docker compose pull` always gets the most recent build. The pinned version tag in GitHub Actions (0.3.0) still provides an explicit version reference in the registry.
