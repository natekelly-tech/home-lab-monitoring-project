## Session 1 — Network Infrastructure Upgrade: Cisco Catalyst 1200 Switch

### Session Overview
Replace unstable WiFi-based lab network with wired switch infrastructure before continuing with Cloudflare tunnel and Docker.

### Problem Statement
The Lab PC (VirtualBox host) was connecting via motherboard WiFi. Running VirtualBox over WiFi caused frequent connection drops, taking down the Ubuntu Server VM and Flask API. A stable wired network was required as a foundation.

### Hardware Introduced

| Device | Model | Role |
|--------|-------|------|
| Managed Switch | Cisco Catalyst 1200-8T-D | Core lab switch |

### Steps Completed

**1. Initial switch access**
- Temporarily set Main PC ethernet adapter to `192.168.1.100/24` to match switch factory default (`192.168.1.254`)
- Accessed web GUI at `https://192.168.1.254`, changed default credentials

**2. Management IP configuration**
- Assigned static IP `192.168.x.x / 255.255.x.x` to switch
- Default gateway: `192.168.x.x` (Speedport), DNS: `1.1.1.1`
- Saved to startup-config, reconnected via new IP

**3. Physical port layout**

| Port | Device |
|------|--------|
| Port 1 | Speedport Smart 4 (uplink) |
| Port 2 | Main PC |
| Port 3 | Lab PC |

**4. Lab PC — WiFi to wired migration**
- Connected Lab PC to switch port 3
- Updated VirtualBox bridged adapter from WiFi NIC to wired ethernet NIC
- Set Lab PC ethernet to static `192.168.x.x/x` (preserving existing address)
- Disabled WiFi adapter on Lab PC

**5. Ubuntu Server VM — connectivity restored**
- VM was showing `[FAILED] cloudflared.service` and 1m 55s network wait on boot
- Fixed by repointing VirtualBox bridged adapter to wired NIC
- Disabled unconfigured cloudflared service temporarily:

```bash
sudo systemctl disable cloudflared
sudo systemctl stop cloudflared
```

## Session 2 — Containerisation: Docker + DockerHub

### Session Overview
Containerise the Flask API using Docker, replace systemd with Docker Compose, and publish the image to DockerHub.

### What Was Accomplished

**1. Docker Engine installed on Ubuntu Server VM**
- Installed from Docker's official apt repository (not snap, not docker.io)
- Version: Docker 29.4.0, Compose v5.1.3
- Added `nate` to `docker` group to avoid requiring sudo

**2. Production Dockerfile written (multi-stage)**
- Base image: `python:3.12-slim` — Debian-based, avoids musl libc issues of alpine
- Stage 1 (builder): installs dependencies into a venv, discarded after build
- Stage 2 (runtime): copies venv + app code only — no pip, no build tools in final image
- Non-root user `appuser` (uid 1001) — limits blast radius if exploited
- `iputils-ping` + `libcap2-bin` installed; `cap_net_raw` set on ping binary via `setcap` — minimum privilege for ICMP checks
- HEALTHCHECK hitting `/` every 30s
- JSON structured logging via `python-json-logger`

**3. Gunicorn replaces Flask dev server**
- Flask dev server is single-threaded and exposes a live debugger over the network
- Gunicorn: 3 workers (`(2 × vCPU) + 1`), timeout 120s (accounts for ProtonVPN slow checks)
- Access logs output as JSON to stdout — captured by Docker log driver

**4. docker-compose.yml written**
- `network_mode: host` — required for ping checks to reach LAN devices
- `cap_add: NET_RAW` — grants ICMP capability at runtime
- `restart: unless-stopped` — replaces systemd's auto-restart
- JSON log driver with rotation (10MB × 3 files)

**5. config.py trimmed for OPSEC**
- Removed Main PC (192.168.2.100) and Speedport Router (192.168.2.1) from monitored services
- Personal dev machine and home router should not be exposed in a public API response
- Remaining: Google, Cloudflare DNS, Lab PC, Ubuntu Server

**6. systemd service retired**
- `homelab-monitor.service` stopped and disabled permanently
- Docker Compose now owns the process lifecycle

**7. Image pushed to DockerHub**
- Tags pushed: `auxcon/labwatch-api:0.1.0` and `auxcon/labwatch-api:latest`
- Public URL: https://hub.docker.com/r/auxcon/labwatch-api

**8. All files committed to GitHub**
- `backend/Dockerfile`, `backend/gunicorn.conf.py`, `backend/docker-compose.yml`
- `backend/.dockerignore`, `backend/requirements.txt`, `backend/app/main.py`

### Issues Resolved

| Issue | Root Cause | Resolution |
|-------|-----------|------------|
| Ping checks showing down | `python:3.12-slim` has no ping binary | Added `iputils-ping` to Dockerfile |
| Ping fails for non-root user | `cap_net_raw` not set on binary | Added `setcap cap_net_raw+ep /usr/bin/ping` in Dockerfile |
| systemd crashing loop | Port 8080 already owned by Docker container | Disabled systemd service |
| git push rejected | Remote had commits not in local copy | `git pull --rebase` then push |

### Key Concepts Learned
- **Multi-stage builds** — builder stage installs deps and is discarded; runtime stage is lean and contains no build tooling
- **Linux capabilities** — `--privileged` grants everything; `cap_net_raw` grants only raw socket access for ping; minimum necessary privilege
- **Container networking** — bridge networking isolates containers from the physical LAN; host networking shares the host's network stack
- **WSGI servers** — Flask's dev server is not production-safe; Gunicorn manages worker processes and handles OS signals correctly
- **Structured logging** — JSON log lines are machine-readable and queryable by log aggregators; plain text is not
- **Reproducible builds** — pinned dependency versions ensure every build produces identical results
