## Key Concepts Learned
- **ISP port blocking** — residential internet providers commonly block incoming connections; enterprise solutions use tunneling to bypass this
- **Cloudflare Tunnel** — outbound tunnel from server to Cloudflare edge network; traffic enters via Cloudflare instead of directly through router; real enterprise pattern used by many companies
- **Dynamic DNS** — keeps a domain pointed at a home IP address even when the ISP changes it; router can auto-update via DuckDNS API
- **Domain registrar vs DNS provider** — Porkbun registers the domain, Cloudflare manages the DNS; these are separate concerns and can be split between providers
- **Nameserver delegation** — by pointing Porkbun nameservers to Cloudflare, all DNS for auxcon.dev is now managed through Cloudflare's dashboard

## Session 2 — Network Infrastructure Upgrade: Cisco Catalyst 1200 Switch Installation

**Date:** 2026-04-18
**Session focus:** Replace unstable WiFi-based lab network with wired switch infrastructure

---

### Problem Statement

The Lab PC (VirtualBox host) was connecting to the home network via motherboard WiFi. Running VirtualBox on top of a WiFi adapter caused frequent connection drops, which took down the Ubuntu Server VM and the Flask API with it. This made the entire monitoring stack unreliable. A wired network was needed as a proper foundation before continuing with Cloudflare tunnel setup and Docker.

---

### Hardware Introduced

| Device | Model | Role |
|--------|-------|------|
| Managed Switch | Cisco Catalyst 1200-8T-D | Core lab switch |

The Catalyst 1200-8T-D is an 8-port gigabit managed switch (SMB tier). Management is via web GUI at a static IP. It supports VLANs, port management, and CLI — relevant for the planned VLAN segmentation in a future phase.

---

### Steps Completed

**1. Initial switch access (direct-connect method)**

Rather than plugging the switch into the Speedport and hunting for a DHCP-assigned IP, the direct-connect method was used:

- Temporarily set Main PC ethernet adapter (192.168.1.100/24) to be on the same subnet as the switch factory default (192.168.1.254)
- Connected Main PC directly to switch via ethernet
- Accessed web GUI at `https://192.168.1.254`
- Default credentials: `cisco` / `cisco` — forced password change on first login
- New admin credentials set and saved to startup-config

**2. Management IP configuration**

Assigned a permanent static IP to the switch on the home network:

- Navigated to IPv4 Configuration → IPv4 Interface
- Added new static entry: `192.168.2.2 / 255.255.255.0`
- Set default gateway: `192.168.2.1` (Speedport) via IPv4 Static Routes (0.0.0.0/0)
- Set DNS: `1.1.1.1` (Cloudflare)
- Saved running-config to startup-config

After applying the new IP the browser session dropped as expected (subnet change). Reverted Main PC ethernet adapter to DHCP, connected switch to Speedport on port 1, reconnected to switch at `https://192.168.2.2`.

**3. Physical port layout**

| Port | Device |
|------|--------|
| Port 1 | Speedport Smart 4 (uplink) |
| Port 2 | Main PC (development machine) |
| Port 3 | Lab PC (VirtualBox host) |

**4. Lab PC — WiFi to wired migration**

- Connected Lab PC to switch port 3 via ethernet cable
- Updated VirtualBox bridged adapter from WiFi NIC to wired ethernet NIC
- Set Lab PC ethernet adapter to static IP `192.168.2.123/24` (preserving original address, avoiding config.py changes)
- WiFi adapter disabled on Lab PC — no longer needed

**5. Ubuntu Server VM — connectivity restored**

- VM was showing `[FAILED] cloudflared.service` on boot and a 1m 55s network wait timeout — both caused by the unconfigured cloudflared service and the bridged adapter previously pointing at WiFi
- After fixing the bridged adapter, VM obtained network correctly
- Disabled cloudflared service temporarily (tunnel not yet configured):

```bash
sudo systemctl disable cloudflared
sudo systemctl stop cloudflared
```
