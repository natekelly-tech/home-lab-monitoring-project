## Session 1 -- Issues Encountered and Resolved

### Tab layout colorScheme error
Old Expo template _layout.tsx referenced Colors.light.tint and Colors.dark.tint. New theme is a flat object. Fixed by removing useColorScheme hook entirely and wiring tab bar colors directly to Colors.accent and Colors.textDisabled.

### API response field mismatch
API returns results array under "results" key, not "services". Fixed by updating the parsing line to: data.results ?? data.services ?? []. Also fixed item.url to item.target to match the API response shape.

### SafeAreaView deprecation
Replaced React Native's built-in SafeAreaView with react-native-safe-area-context version in index.tsx and settings.tsx.

### AbortSignal.timeout not supported
React Native's JS engine does not support AbortSignal.timeout(). Replaced with AbortController + setTimeout pattern in both settings.tsx and welcome.tsx.

### Tunnel connectivity
npx expo start --tunnel was the only reliable way to connect the phone during this session. Local network mode consistently failed with Java IO exception. USB debugging via adb worked for device detection but was slow to start. Installed @expo/ngrok@^4.1.0 to fix a stale ngrok version error.

### Onboarding routing -- extended debugging
The welcome screen routing took significant time to get right. Root cause: expo-router's file-based routing auto-discovers welcome.tsx but the onboarding redirect was firing before the navigator was ready. Solution: check moved to app/_layout.tsx using a useEffect that fires after mount, with initialRouteName set to (tabs) so the app always lands on the dashboard unless redirected. welcome.tsx was also found to be missing from the app/ folder at one point, having not been saved there correctly.

### Backend config not updating
After editing config.py, the local docker build on EC2 was used but docker-compose.yml pulls from DockerHub, not local images. The compose pull kept getting the old image. Fixed by understanding the correct workflow: push to GitHub, tag to trigger Actions, then compose pull on EC2. The local build was a detour.

## Session 1 -- Decisions Made

### Expo stays
Confirmed Expo is the right choice. The "it looks like a template" feeling came from template code, not the framework. Production EAS builds have zero Expo branding. Shopify, Coinbase, Discord all use React Native/Expo.

### api.auxcon.dev is never multi-tenant
Explicitly decided: the public backend is a demo and personal reference implementation only. Never SaaS, never multi-tenant. Users run their own. This keeps the product story clean and avoids compliance complexity.

### k8s framing
Kubernetes is not just a course deliverable. It is the delivery format for the self-hostable reference implementation at scale. The k8s/ manifests in the repo are what a data center engineer looks at when deciding whether to trust the software.

## Session 2 -- Issues Encountered and Resolved

### Unmatched Route -- file in wrong directory
After creating the detail screen file, tapping a card produced "Unmatched Route / Page could not be found". Root cause: file was created at project root as service/[name].tsx instead of app/service/[name].tsx. expo-router only discovers routes inside app/. Resolution: moved file to correct location using PowerShell. During the move, Remove-Item deleted the file before Move-Item completed. File had to be recreated from scratch.

Lesson: verify file placement with Get-ChildItem before testing on device:
  Get-ChildItem -Recurse -Filter "*.tsx" | Where-Object { $_.FullName -notlike "*node_modules*" } | Select-Object FullName

### Router push encoding
Initial implementation used a template literal: router.push(`/service/${item.name}`). Breaks for service names with spaces. Switched to params object form before testing:
  router.push({ pathname: '/service/[name]', params: { name: item.name } })

## Session 2 -- Decisions Made

### Re-fetch on detail screen mount, not nav params
Service data object is not passed through navigation params. Detail screen re-fetches GET /status independently. Params are URL-serialized which degrades for complex objects. Re-fetching also keeps the view current.

### Deterministic mock data, not random
Math.random() would produce different numbers on every render, making the screen look broken in development and demos. PRNG seeded from service name character codes ensures same service always shows same numbers.

### No charting library
react-native-chart-kit and similar require native module setup, rebuild time in Expo Go, and App Store review surface area. Flex-row sparkline is sufficient for mocked data. Revisitable in Phase 8.

## Session 3 -- Issues Encountered and Resolved

### Named vs default export crash (recurring)
theme.ts has no default export. Every import that used `import Colors from '...'` crashed at runtime with "Cannot read property 'bg' of undefined". Same issue hit in app/(tabs)/_layout.tsx and app/backends.tsx. Fix: always use `import { Colors } from '...'` with curly braces.

Lesson: if a module has no `export default`, it must be imported with destructuring. TypeScript may not catch this at compile time in all configurations.

### expo-router auto-discovery vs Stack.Screen conflict
Adding an explicit Stack.Screen entry for 'backends' in _layout.tsx while expo-router also auto-discovered backends.tsx produced a "Too many screens defined. Route 'backends' is extraneous" warning. Resolution: suppress the native header from inside the screen file using `<Stack.Screen options={{ headerShown: false }} />` as the first child of the return, and remove the explicit entry from _layout.tsx. ## Session 3 -- Decisions Made

### Stack.Screen inside screen file, not in _layout.tsx for auto-discovered routes
expo-router v3 auto-discovers all .tsx files inside app/. Explicit Stack.Screen entries in _layout.tsx for these routes create conflicts. The correct pattern for header suppression on auto-discovered routes is to place Stack.Screen as the first JSX child inside the screen's return statement.

### Demo backend not deletable
The demo backend is a product guarantee. No matter what a user does, they can tap it to get back to a working state. Deleting it would require handling the empty-list edge case with no benefit.

### Save and Activate on backend add
Matches conventions in VPN apps, proxy switchers, and similar tools. If the user does not want the new backend active, they can immediately tap another in the list.

### BACKEND section in settings.tsx not removed yet
The legacy manual URL editor in settings.tsx is now redundant since the backends screen handles this properly. Left in place because Dollahite has seen the settings screen as-is and removing it is a cleanup task, not a functional change. Flagged as a pre-App-Store cleanup item.
