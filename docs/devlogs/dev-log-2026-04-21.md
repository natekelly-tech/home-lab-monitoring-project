## What Was Done

### 1. Rate Limiting (flask-limiter)

Added flask-limiter 3.9.0 to the Flask API with a 60 requests per minute default limit applied globally to all routes.

The public API at `api.auxcon.dev` had no rate limiting. Without it, any client could make unlimited requests — enabling brute force probing, denial of service via resource exhaustion, or aggressive scraping.

- Added `flask-limiter==3.9.0` to `backend/requirements.txt`
- Added imports and limiter initialisation in `backend/app/main.py` directly after `app = Flask(__name__)`
- `default_limits=["60 per minute"]` applies globally — individual routes unchanged

**Known limitation:** Gunicorn runs 3 worker processes. flask-limiter in-memory storage tracks counts per worker independently. A client distributed across workers could reach 180rpm before being blocked. Correct fix is a Redis shared backend — deferred to Kubernetes phase where it will run as a sidecar. TODO comment left in `main.py`.

Deployed to production via GitHub Actions.

### 2. System Updates

Ran `sudo apt upgrade -y` on EC2. Six packages upgraded: docker-model-plugin, libcap2, libcap2-bin, libpam-cap, libntfs-3g89t64, ntfs-3g. Five were LTS security updates. fail2ban restarted automatically during upgrade and confirmed active.

### 3. Certbot TLS Certificate

Installed Certbot with the Cloudflare DNS plugin and issued a Let's Encrypt TLS certificate for `api.auxcon.dev`.

HTTP-01 challenge was not viable — `api.auxcon.dev` routes through Cloudflare Tunnel, so Let's Encrypt cannot reach EC2 on port 80. DNS-01 proves domain ownership via a temporary TXT record in Cloudflare DNS instead. No inbound HTTP required.

**Cloudflare API token:**
- Name: `certbot-auxcon`
- Scope: `auxcon.dev` DNS Write only
- Expiry: 1 year

**Certificate:**
- Domain: `api.auxcon.dev`
- Key type: ECDSA
- Auto-renewal: systemd timer configured by Certbot

Cloudflare Tunnel currently handles TLS termination at the edge, the cert is available for direct port 443 serving if the tunnel is bypassed.

### 5. Trivy Scans

First scan since migrating to EC2. Scanned both the container image and the EC2 host OS as separate threat surfaces.

**Container scan — auxcon/labwatch-api:latest**

CRITICAL: 0. Secrets: 0.

Two Flask CVEs found with fixes available:
- CVE-2025-47278 — session signing fallback key vulnerability. Fixed in 3.1.1.
- CVE-2026-27205 — information disclosure via improper session data caching. Fixed in 3.1.3.

All other HIGH/MEDIUM findings in ncurses, systemd, glibc, util-linux — no fixes available upstream. Documented and accepted.

**EC2 host OS scan**

Full filesystem scan (`trivy rootfs /`) timed out, single CPU cannot handle scanning all files including Docker overlay layers. Rerun with `trivy os --scanners vuln /` to scan installed OS packages only.

Key findings:
- Kernel: Hundreds of LOW/MEDIUM CVEs — upstream blocked, no fixes from Ubuntu. Normal for production kernels.
- snapd / Amazon SSM agent: CRITICAL/HIGH in Go binaries — managed by Canonical and AWS. Not actionable.
- cloudflared: CRITICAL CVE-2026-33186 — gRPC authorization bypass. Attempted upgrade — already on latest (2026.3.0). No patch from Cloudflare yet. Upstream blocked.
- Secrets scanner: Flagged SSH host keys, TLS cert private key, firmware update service key — all legitimate system files in correct locations. Not real findings.
- System Python packages: Multiple HIGH/MEDIUM — installed via apt, managed by Ubuntu. Not actionable.

No CRITICAL vulnerabilities that are actionable. Attack surface locked down via UFW, fail2ban, and key-only SSH. cloudflared finding will remediate automatically when Cloudflare releases a patched package.

## Decisions Made

**flask-limiter in-memory over Redis:** Redis is the correct fix but introduces a stateful service dependency before Kubernetes. Kubernetes is the right place to bring Redis in as a managed sidecar. Adding it now is premature.

**DNS-01 over HTTP-01 for Certbot:** Domain routes through Cloudflare Tunnel. HTTP-01 cannot reach EC2 on port 80. DNS-01 via the Cloudflare plugin is the only viable approach.

**Removed root-level Dockerfile and requirements.txt:** `backend/` versions are authoritative. Root-level copies served no purpose and created confusion about which files CI/CD actually uses.
