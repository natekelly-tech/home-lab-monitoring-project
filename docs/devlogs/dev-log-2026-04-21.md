## 1. Rate Limiting (flask-limiter)

**What we did:**
Added flask-limiter 3.9.0 to the Flask API with a 60 requests per minute default limit applied globally to all routes.

**Why:**
The public API endpoint at `api.auxcon.dev` had no rate limiting. Without it, any client could make unlimited requests — enabling brute force probing, denial of service via resource exhaustion, or aggressive scraping. This is a standard security baseline for any public-facing API.

**Implementation:**
- Added `flask-limiter==3.9.0` to `backend/requirements.txt`
- Added imports and limiter initialisation in `backend/app/main.py` directly after `app = Flask(__name__)`
- Left individual routes unchanged — `default_limits=["60 per minute"]` applies globally

**Known limitation documented:**
Gunicorn runs 3 worker processes. flask-limiter's default in-memory storage tracks request counts per worker independently, not across all workers. This means a client making requests distributed across workers could hit up to 180rpm before being blocked. The correct fix is a Redis backend shared by all workers.

**Decision to defer Redis:**
Adding Redis as a standalone container now would introduce a new stateful service dependency before Kubernetes. The right architectural home for Redis is as a sidecar service in Kubernetes, where it can be managed, scaled, and observed properly. A TODO comment was added in `main.py` documenting this limitation and deferral decision.

**Result:** Deployed to production. GitHub Actions built and pushed new image. `docker compose pull && docker compose up -d` deployed to EC2.

## 3. Certbot TLS Certificate

**What we did:**
Installed Certbot with the Cloudflare DNS plugin and issued a valid Let's Encrypt TLS certificate for `api.auxcon.dev`.

**Why HTTP-01 challenge would not work:**
The standard Certbot HTTP-01 challenge works by having Let's Encrypt send an HTTP request to port 80 of the domain being verified. Because `api.auxcon.dev` routes through Cloudflare Tunnel rather than directly to the EC2 IP, Let's Encrypt cannot reach the EC2 server on port 80. The request would be intercepted by Cloudflare's edge.

**Why DNS-01 was the right approach:**
The DNS-01 challenge proves domain ownership by creating a temporary TXT record in the domain's DNS zone. Because the domain is managed on Cloudflare and we have a scoped API token with DNS write permissions, Certbot can create and delete the verification record automatically. No inbound HTTP is required.

**Cloudflare API token created:**
- Name: `certbot-auxcon`
- Scope: `auxcon.dev` DNS Write only (least privilege — no account-level access)
- Expiry: 1 year
- Stored at: `/etc/letsencrypt/cloudflare.ini` with permissions `chmod 600`

**Certificate details:**
- Domain: `api.auxcon.dev`
- Key type: ECDSA
- Valid until: 2026-07-20
- Auto-renewal: configured via systemd timer installed by Certbot

**Current state:**
The certificate exists on disk at `/etc/letsencrypt/live/api.auxcon.dev/`. It is not currently being served directly by the application — Cloudflare Tunnel handles TLS termination at the edge. The certificate is available for direct HTTPS serving on port 443 if the tunnel is bypassed, which is the scenario Dollahite had in mind as a learning exercise.

## 4. Repo Structure Cleanup

**What we did:**
Removed redundant root-level `Dockerfile` and `requirements.txt` from the repository.

**Why they were redundant:**
The repository root contained a `Dockerfile` and `requirements.txt` that were copies of the files inside `backend/`. The `backend/` versions are the ones used by GitHub Actions CI/CD and Docker builds. The root-level copies served no purpose and created ambiguity about which files were authoritative.

**What was kept:**
`SECURITY.md` at the root was intentionally retained. It is a repository-level security policy document, not a backend artifact, and belongs at the root.

**Note on CI/CD behaviour:**
The `git push` that removed these files did not trigger a Docker build. This is correct — the GitHub Actions workflow only triggers a build when files inside `backend/` change. Removing root-level files is invisible to the build pipeline.

## 5. Trivy Scans

Ran two Trivy scans: a container image scan and an EC2 host OS scan. This was the first scan since migrating to EC2, so scanning both the container and the host as separate threat surfaces was appropriate.

### Container Scan — auxcon/labwatch-api:latest

**Summary:** CRITICAL: 0, Secrets: 0. Clean on both critical metrics.

**Flask vulnerabilities found (actionable):**
- CVE-2025-47278 — Flask 3.1.0 session signing fallback key vulnerability. Fixed in 3.1.1.
- CVE-2026-27205 — Flask information disclosure via improper session data caching. Fixed in 3.1.3.

Both were status `fixed` — meaning a patched version was available. Flask was upgraded to 3.1.3 in `backend/requirements.txt` and deployed to production.

**Everything else — upstream blocked:**
All HIGH and MEDIUM findings were in ncurses, systemd, glibc, util-linux, and related system libraries. No fixes available from Debian. Same category as the previous scan. Documented and accepted.

### EC2 Host OS Scan

The full filesystem scan (`trivy rootfs /`) timed out on the t2.micro — the single CPU cannot handle scanning the entire filesystem including Docker container overlay layers within the default timeout. This is an expected hardware constraint, not a scan failure.

The scan was rerun using `trivy os --scanners vuln /` which scans only installed OS packages rather than all files. This is appropriate for our goal of assessing the host OS package surface.

**Key findings:**

- Kernel: Hundreds of LOW/MEDIUM CVEs in the AWS kernel package. All upstream blocked — no fixes available from Ubuntu. This is normal for production Linux kernels which accumulate CVEs between major update cycles.
- snapd Go binaries: CRITICAL/HIGH findings in the snapd and Amazon SSM agent Go binaries. These are managed by Canonical and AWS respectively. Not actionable.
- cloudflared: CRITICAL CVE-2026-33186 — gRPC authorization bypass in `google.golang.org/grpc`. Attempted upgrade via `sudo apt install --only-upgrade cloudflared` — confirmed already on latest available version (2026.3.0). Upstream blocked, no patch from Cloudflare yet.
- Secrets scanner: Flagged SSH host keys (`/etc/ssh/`), the TLS certificate private key (`/etc/letsencrypt/`), and a firmware update service key (`/var/lib/fwupd/`). These are all legitimate system files sitting exactly where they should be. Not real findings.
- System Python packages: Multiple HIGH/MEDIUM in system Python packages (cryptography, setuptools, urllib3, etc.) installed via apt. These are managed by Ubuntu and not actionable individually.

**Overall host posture:** No CRITICAL vulnerabilities that are actionable. The attack surface is appropriately locked down via UFW, fail2ban, and key-only SSH. The cloudflared grpc finding is noted and will be remediated automatically when Cloudflare releases a patched package.
